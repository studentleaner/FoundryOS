# Validation Mechanics

The OS redefines "Validation". Traditional accelerators permit surveys, interviews, and "opinions" to count as validation. The FoundryOS strictly enforces **Behavioral Evidence** and **Observable Actions**.

## The Philosophy of Validation
- "Talk less, listen more."
- "Measure behavior, not opinions."
- The fundamental rule: **If nothing real happened, nothing is validated.**

## Acceptable Validation Signals
Validation is only achieved when users leave an auditable data trail proving intent.
1. **Unprompted Complaints:** Users actively discussing the pain without solicitation.
2. **Email Submissions:** Providing personal data in exchange for a promised solution.
3. **Willingness to Pay:** High-intent behavior like pre-orders, refundable deposits, or paid pilot contracts.
4. **Time & Attention Span:** Measurable time spent discussing a problem, interacting with a prototype, or navigating a workflow.

## Unacceptable (Kill) Signals
The presence of the following signals indicates the assumption has failed validation:
1. **Polite Interest:** Users claiming an idea "sounds cool."
2. **Feature Creep Before Proof:** Expanding the MVP scope instead of proving the core value loop.
3. **No Behavior Change:** The target persona recognizes the tool but refuses to alter their current process/workflow.
4. **Lack of Organic Urgency:** The problem is deemed a "nice to have," meaning they will endure the pain rather than adopt your solution.

## The Validation Experiment Loop
1. **Define the Signal Upfront:** Before launching, state explicitly what metric constitutes success (e.g., "5% conversion on the pricing tier").
2. **Implement Irreversible Tests:** Create an environment where a real user must complete ONE full journey end-to-end to generate the data point.
3. **Capture the Event:** Instrument the precise flow so success or failure is logged via analytics.
4. **Decide:** Execute the pivot/proceed/kill decision strictly based on the logged evidence.
